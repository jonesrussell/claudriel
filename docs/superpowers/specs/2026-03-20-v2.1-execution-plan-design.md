# v2.1 Core Agent Enhancements: Execution Plan

**Date:** 2026-03-20
**Milestone:** v2.1 — Core Agent Enhancements (28 open issues)
**Approach:** Foundation-first, then parallel feature blast

## Overview

Tackle all 28 open v2.1 issues in ~12 sessions across 3 phases:

1. **Foundation** (sessions 1-4): Fix bugs and stabilize infra so feature work isn't fighting breakage
2. **Parallel Features** (sessions 5-8): Three independent feature chains in worktree branches
3. **Integration + Polish** (sessions 9-12): Long-context support and frontend editorial polish

## Phase 2: Parallel Feature Tracks (11 issues)

Three independent chains. Each gets its own worktree branch. Can be worked in any order or in parallel sessions.

### Track A: Judgment Rules (`feat/judgment-rules`)

Sequential chain: #292 -> #295 -> #298 -> #301

| Session | Issue | Title | Depends on |
|---------|-------|-------|------------|
| 5 | #292 | Judgment Rules System | Sessions 1-3 complete (no agent infra dependency) |
| 6 | #295 | JudgmentRule entity + GraphQL schema | #292 |
| 7 | #298 | Judgment rules internal API + agent tool | #295 |
| 8 | #301 | Judgment rules prompt injection into agent | #298 |

Key design decisions (from spec `docs/roadmap/specs/01-judgment-rules.md`):
- JudgmentRule entity with tenant scoping, confidence scoring
- Max 100 rules/tenant, 500 char limit, 2000 token prompt budget
- User corrections = confidence 1.0, agent suggestions = 0.7
- Rules ranked by application count for prompt injection

### Track B: Agent Continuation (`feat/agent-continuation`)

Sequential chain: #305 -> #308 -> #310

| Session | Issue | Title | Depends on |
|---------|-------|-------|------------|
| 5 | #305 | Agent Continuation Mechanism | Sessions 1-3 complete (no agent infra dependency) |
| 6 | #308 | Agent turn tracking and continuation events | #305 |
| 7 | #310 | Continue button UI + turn limit settings | #308 |

Key design decisions (from spec `docs/roadmap/specs/08-agent-continuation.md`):
- Configurable per-task-type turn budgets (quick_lookup: 5, research: 40, general: 25, etc.)
- `needs_continuation` event at limit, frontend "Continue?" button
- Daily ceiling per tenant (default 500 turns/day)
- ChatSession extended with turns_consumed, task_type, continued_count

- 12 new HMAC-protected internal API endpoints + agent tools

- Batched rollout reduces risk and testing burden

### Parallelism Strategy

- Each track gets a git worktree branch
- Tracks share no code paths (Judgment Rules = new entity/API, Continuation = session/chat, Tool Richness = internal API endpoints)
- Merge each track to main after its full chain completes and passes CI
- If merge conflicts arise, they'll be minimal (service provider registrations)
- If a track is not ready when Phase 3 starts, merge the completed tracks and begin Phase 3; the remaining track merges later
- Tracks A and B can start after session 3 (no agent infra dependency); Track C requires session 4 (#379, #380)

## Acceptance Criteria

Each issue's acceptance criteria is defined in its GitHub issue description. A session is done when:
- All issues in the session pass their GitHub-defined acceptance criteria
- Tests pass (existing + new)
- CI is green
- For worktree tracks: branch is ready to merge (no conflicts with main)

## Testing Strategy

Each session follows TDD per the existing specs:
- **PHP integration tests:** Entity persistence, API auth, tenant isolation, response formats
- **Python agent tests:** Tool registration, HMAC auth, error handling
- **Playwright UI tests:** Rule management, continuation button, tool-backed answers (Phase 2+3)
- **Structured logging:** Audit trails for rule creation, turn tracking, tool calls

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| Worktree merge conflicts | Tracks touch different subsystems; conflicts limited to service provider |
| Agent prompt too large (rules + tools) | 2000 token rule budget + 4000 token tool budget = 6000, well within limits |
| Frontend polish scope creep | CSS-only constraint from editorial polish plan |
| #378 fix more complex than expected | Entity registration pattern is well-documented in waaseyaa-entity-wiring.md |
| One feature track blocks Phase 3 | Phase 3 can proceed with completed tracks; blocked track merges independently |
| Frontend polish file conflicts | Verify shared component overlap before parallelizing session 12 |
